## Implementation shape

I would make the first AI SDK chat state version deliberately plain. The AI SDK chat state system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make AI SDK chat state enterprise software. It makes the work inspectable.

A useful AI SDK chat state implementation question is where the record is created. If the AI SDK chat state record is created only after success, it will miss the interesting failures. I would create the AI SDK chat state record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The AI SDK chat state tradeoff is convenience versus what the browser, content system, and deployment pipeline each own. If the AI SDK chat state system is too strict, people route around it. If the AI SDK chat state system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For AI SDK chat state, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful AI SDK chat state shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The AI SDK chat state pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The AI SDK chat state metric is route coverage, feed validity, content validation, accessibility checks, and build time. That is more useful than asking whether AI SDK chat state "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For AI SDK chat state, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those AI SDK chat state numbers usually tell the truth before the happy-path dashboard does.

If the AI SDK chat state system cannot answer those questions, I would keep the design smaller. Small AI SDK chat state systems with honest receipts are easier to improve than large systems that only report success.
