That distinction matters because vector search can be built around a lot of different ideas. Similarity might mean wording, intent, permission, recency, or usefulness in the next action. Those are not the same thing. If the product team has not chosen one, the index is doing more philosophical work than it should.

I have seen search systems that look impressive in a demo and then fall apart the second a user asks a slightly different question. The fix is rarely more vectors. Usually it is a better product question. What is the user trying to do? What kind of evidence should rank first? What should never be surfaced even if it looks close?

The part I keep coming back to is that vector search is not a replacement for information design. It is a retrieval method. The product still has to decide whether the problem is finding documents, answering questions, or reducing friction in a workflow. Once that is clear, the index becomes useful instead of theatrical.
