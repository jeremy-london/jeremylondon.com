Tailwind CSS v4 moved more of the system into CSS itself, which made configuration feel less separate from styling. I liked that because I have never loved configuration that lives too far away from the thing it changes. Every extra layer between the decision and the style makes the work feel more abstract than it needs to be.

What v4 did well, in my view, was reduce some of that translation. If the token, the variable, and the rule live closer together, I can read the stylesheet without switching mental models every few lines. That makes the code easier to reason about, especially when the system is large and there are a lot of design decisions hiding behind utility classes.

At the same time, I do not think pulling more into CSS is automatically a win. It only helps if the team keeps the names honest and the overrides deliberate. A stylesheet can become a dumping ground just as easily as any other config file. The nice part is that it now looks more like the thing it is controlling, so the mess is easier to spot.

That is the change I cared about. The framework moved some configuration back into a place I already think about when I work on UI. That lowers the translation cost and makes the design decisions easier to inspect.

I still want discipline around it, though. A better place for configuration does not remove the need to know why the value exists.
