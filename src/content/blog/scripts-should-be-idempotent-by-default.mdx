The best project scripts can run twice without making the second run weird.

The interesting part of scripts is not the name. It is the system pressure around The best project scripts can run twice without making the second run weird. Once that pressure shows up for scripts, the topic stops being a feature and starts being an interface between people, models, code, and evidence.

## The unit of work

I would start by naming the unit of work for scripts. In developer tools, the scripts unit is usually smaller than the product surface and larger than a function call. For scripts, it might be one agent step, one retrieval request, one classification decision, one model route, one sync attempt, or one evaluation case.

A useful scripts unit has four properties:

- an input the system can store
- a decision the system can explain
- evidence the next component can inspect
- a boundary that says who or what is allowed to act

For scripts, the boundary is what the tool may change without asking. That boundary should be visible in the scripts data, not only in a policy document. If a scripts request crosses the boundary, the trace should show it.

## Data flow

The simple scripts flow looks like this:

~~~text
request -> context builder -> decision point -> action or answer -> verifier -> receipt
~~~

The context builder gathers the minimum useful state for scripts. The scripts decision point chooses a route, tool, model, policy, or user-facing behavior. The verifier checks the output against the contract. The receipt records enough evidence for review.

That receipt is the difference between a scripts demo and an operable system. A demo can rely on a person watching closely. A scripts system needs to explain itself after the person has moved on.

A scripts request envelope might look like this:

~~~json
{
  "input": "user or system request",
  "context": ["retrieved document", "current state", "policy hint"],
  "decision": "selected route or action",
  "evidence": ["why this route was allowed"],
  "limits": { "costCents": 12, "timeoutMs": 8000 },
  "requiresApproval": false
}
~~~

The schema is deliberately plain. Fancy schemas do not save unclear scripts systems. The useful scripts work is deciding which fields are required, which are advisory, and which block the run when missing.

## Implementation sketch

I would implement the first scripts pass as a narrow pipeline rather than a general platform. Start scripts with a request envelope, make the route explicit, attach evidence, run the cheapest deterministic checks first, then escalate to semantic judgment only when the deterministic checks cannot answer the question.

~~~ts
type StepResult = {
  route: string
  output: unknown
  evidence: string[]
  deterministicChecks: string[]
  semanticReview?: string
  accepted: boolean
}
~~~

For scripts, this shape matters because it keeps "the model said so" out of the acceptance path. The model can propose. The scripts system still has to decide whether the proposal is usable under the current boundary.

## Failure modes

The main scripts failure mode is the tool saves a minute while hiding the mistake that costs an afternoon. It rarely announces itself cleanly. The model may return a good-looking answer. The UI may render. The job may exit zero. The scripts failure is that the next component cannot tell whether the result should be trusted.

I would test at least five scripts cases:

- missing context
- stale context
- valid output with insufficient evidence
- correct output under the wrong permission boundary
- high-confidence output that fails a deterministic check

That last scripts case matters. A confident scripts answer is not the same thing as a verified result. If scripts touches state, policy, spend, or user trust, confidence should never be the only acceptance criterion.

## Deployment shape

I would not deploy scripts as one giant agent with every tool and every permission. I would split the system by risk class. Low-risk read-only work can move quickly. State-changing work needs stronger checks. Credentialed work needs explicit scope, expiration, and audit. For scripts, anything irreversible should have a human stop unless the surrounding domain has already proven a safer automatic path.

That gives the router real work to do. For scripts, it should not only choose small model or big model. It should choose a risk path:

- read-only answer
- draft-only action
- reversible write
- privileged action
- escalation

Each scripts path has a different budget, verifier, and approval requirement. This is the part that makes scripts feel less magical and more like infrastructure. The model is still important, but the scripts route determines what kind of trust the system is asking for.

## Measurement

The measurement surface for scripts is time to diagnose, idempotent reruns, exit-code accuracy, and log usefulness. I would track scripts per route or case type, not only as a global average. Global averages hide the exact class of scripts problems that make these systems feel unpredictable.

A simple eval table can carry a lot of weight:

~~~text
case_id | route | expected | actual | verifier | latency_ms | cost_cents | accepted
~~~

The point is not to build an eval cathedral for scripts. The point is to preserve enough scripts structure that regressions can be clustered. If failures cluster around one route, fix the route. If scripts failures cluster around one policy boundary, fix the boundary. If scripts failures cluster around one model, stop pretending the prompt is the whole system.

## Operational concerns

The operational question for scripts is how this behaves when the happy path stops being representative. What happens when context is missing? What happens when the verifier times out? What happens when the cheap scripts route becomes unreliable but the expensive route would blow the budget?

For scripts, I would put those answers in configuration, not in a heroic prompt. A small scripts policy file that names timeouts, budgets, fallback routes, and escalation behavior will beat a clever paragraph in the system prompt every time.

## Scaling behavior

At small scale, a human can inspect weird scripts cases by hand. At medium scale, the scripts system needs sampling, dashboards, and escalation. At larger scale, it needs budgets and revocation. The design changes because the bottleneck changes.

Early on, the bottleneck is usually correctness. Later it becomes review capacity. After that it becomes trust: can teams change the scripts system without being surprised by the downstream behavior?

For scripts, I would keep the first architecture boring. Store the request envelope. Store the decision. Store the evidence. Run deterministic checks where possible. Use semantic checks where necessary. Escalate when the verifier cannot decide. That is not glamorous, but it gives the scripts system a spine.

## Where I land

Scripts deserves depth when it changes architecture, not when it merely sounds current. The scripts implementation questions are concrete: what is the unit of work, what evidence moves with it, who can approve it, how does it fail, and what metric catches the failure before users do?

If the scripts post answers those questions, it teaches something. If the scripts post only says the topic matters, it is just a longer version of the headline.
