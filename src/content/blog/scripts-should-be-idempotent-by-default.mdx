The best project scripts can run twice without making the second run weird. The point is not to make scripts sound bigger than it is. It is to name the decision it changes.

## The Practical Part

Scripts gets interesting when it changes a real workflow. The example I would use is the best project scripts can run twice without making the second run weird, because it gives the reader something to inspect instead of a broad claim to accept.

## The Edge Case

The failure mode for scripts is the tool saves a minute while hiding the mistake that costs an afternoon. For scripts, that can look fine from a distance, so the page ships, the demo runs, or the team agrees with itself while the next person still cannot tell what actually happened.

## What I Want To See

The check for scripts is the command output that tells the user what happened. For scripts, that is enough to separate a useful note from a long one.

I do not want scripts to sound more certain than the evidence allows. Make the decision visible, then stop.
