Dev Mode made design-system drift easier to see because the design file became more inspectable by developers.

I would keep figma small here. The useful part is Dev Mode made design-system drift easier to see because the design file became more inspectable by developers. That is enough material for a figma note, not a manifesto.

The figma mistake is treating the observation as if it needs a complete theory around it. In practice, design systems work usually gets better when the first version names the boundary and then stops. For figma, that boundary is which decisions belong in tokens, components, product copy, or one-off layout.

The figma check I would use is simple: can the next person see what changed and why it matters? If the answer is yes, the figma post has done its job. If the answer is no, adding three more figma paragraphs will not fix it.

The thing I would not add to figma is a fake architecture section. A short figma note can still be technical if it names the pressure clearly: who is acting, what state is being touched, and what evidence would make the claim falsifiable. That is enough for figma. The rest of figma can wait until there is a real example to inspect.

I am leaving figma as a short note because the idea is useful at this size. A longer figma version would need a real system diagram, a failure example, and evidence. Without those, more length would just be decoration.
