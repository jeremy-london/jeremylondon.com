Conflict resolution belongs in the product experience, not only in the replication layer.

I would keep sync small here. The useful part is Conflict resolution belongs in the product experience, not only in the replication layer. That is enough material for a sync note, not a manifesto.

The sync mistake is treating the observation as if it needs a complete theory around it. In practice, local-first software work usually gets better when the first version names the boundary and then stops. For sync, that boundary is what must be true before the server is reachable.

The sync check I would use is simple: can the next person see what changed and why it matters? If the answer is yes, the sync post has done its job. If the answer is no, adding three more sync paragraphs will not fix it.

The thing I would not add to sync is a fake architecture section. A short sync note can still be technical if it names the pressure clearly: who is acting, what state is being touched, and what evidence would make the claim falsifiable. That is enough for sync. The rest of sync can wait until there is a real example to inspect.

I am leaving sync as a short note because the idea is useful at this size. A longer sync version would need a real system diagram, a failure example, and evidence. Without those, more length would just be decoration.
