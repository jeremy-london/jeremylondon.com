Profiling moving into the OpenTelemetry conversation made performance feel closer to traces, metrics, and logs. I liked that because performance was never really separate from observability in the first place. It was just treated as a specialist concern for too long. Once profiling sits in the same place as traces and metrics, the story gets easier to follow.

The value for me is not the profiler itself. It is the context around it. A hot function by itself is a clue. A hot function tied to a customer path, a request ID, and a slow span is something I can work with. That is the difference between knowing a process is expensive and knowing which user journey is paying for the cost.

I have seen performance work turn into guesswork when the profile is detached from the rest of the telemetry. Someone opens a flame graph, sees a suspicious function, and then spends the next hour proving that the function was only expensive in one narrow case. OpenTelemetry makes that investigation cleaner because the signal stays connected to the request story.

That connection matters for AI systems too, which is probably why I keep writing about it. If the model path is slow, I want to know whether it is the prompt, the retrieval, the post-processing, or just the hardware. I do not want a separate performance tool that lives far away from the trace the incident team is already reading.

My basic view is that profiling should be part of the same conversation as logging and tracing. Performance bugs are easier to fix when they are not hidden in a different workflow.
