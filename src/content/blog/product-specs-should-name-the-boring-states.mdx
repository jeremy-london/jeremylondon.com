A product spec that skips loading, empty, error, permission, and rollback states is not finished. I have a strong opinion about this because that is usually where the product turns into guesswork. The happy path is easy to describe. The real product starts when the user waits, gets blocked, retries, or lands in a state nobody remembered to design.

What tends to happen is that engineering invents the missing behavior under pressure and the result quietly becomes the product. That is a bad way to work. The spec should not only describe the screen that appears when everything goes right. It should say what happens when data is missing, when permissions are wrong, when the request is slow, and when the user needs to recover from a partial failure without losing their work.

I think this matters because the boring states are where trust gets built or lost. If an app handles an error badly, users remember that. If it handles a rollback badly, support remembers it. If it hides the loading state and then flashes the wrong content for a second, design now has a bug that looks like a feeling. Those problems are small on paper and loud in practice.

The best specs I have worked from were the ones that made me uncomfortable in a good way. They forced me to answer what should happen if the user comes back later, what should be preserved, and what the interface should say when the system is unsure. That is harder than writing "show spinner" and moving on, but it is also the difference between a sketch and a product.

I keep saying the same thing in different words because it matters: if the spec does not name the boring states, implementation will. And once implementation names them, it usually does so in the fastest possible way, which is rarely the best one.
