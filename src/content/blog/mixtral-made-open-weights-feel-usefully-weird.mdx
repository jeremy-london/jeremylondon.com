## What has to be true

For mistral to be useful, the AI systems builder needs a clear model decision note. That artifact does not have to be fancy. For mistral, it can be a log line, a JSON envelope, a rendered state, a small script output, or a runnable example. What matters for mistral is that it explains what happened without asking the next person to reconstruct intent from vibes.

The implementation detail I care about is ownership. Who creates the artifact? Who reads it? Who is allowed to change it? In model watch, the annoying mistral bugs usually live between those answers. ## Implementation shape

I would make the first mistral version deliberately plain. The mistral system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make mistral enterprise software. It makes the work inspectable.

A useful mistral implementation question is where the record is created. If the mistral record is created only after success, it will miss the interesting failures. I would create the mistral record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The mistral tradeoff is convenience versus what changed in the model versus what changed in the product surface. If the mistral system is too strict, people route around it. If the mistral system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For mistral, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful mistral shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The mistral pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The mistral metric is route changes, eval deltas, latency movement, cost movement, and migration work. That is more useful than asking whether mistral "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For mistral, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those mistral numbers usually tell the truth before the happy-path dashboard does.

If the mistral system cannot answer those questions, I would keep the design smaller. Small mistral systems with honest receipts are easier to improve than large systems that only report success.
