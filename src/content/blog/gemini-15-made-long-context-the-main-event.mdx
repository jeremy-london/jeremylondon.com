## What has to be true

For google to be useful, the AI systems builder needs a clear model decision note. That artifact does not have to be fancy. For google, it can be a log line, a JSON envelope, a rendered state, a small script output, or a runnable example. What matters for google is that it explains what happened without asking the next person to reconstruct intent from vibes.

The implementation detail I care about is ownership. Who creates the artifact? Who reads it? Who is allowed to change it? In model watch, the annoying google bugs usually live between those answers. A feature works, but the trace is missing. A model returns something valid for google, but the caller cannot tell whether it is safe. A script exits zero for google, but the output hides the one fact the maintainer needed.

## Implementation shape

I would make the first google version deliberately plain. The google system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make google enterprise software. It makes the work inspectable.

A useful google implementation question is where the record is created. If the google record is created only after success, it will miss the interesting failures. I would create the google record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The google tradeoff is convenience versus what changed in the model versus what changed in the product surface. If the google system is too strict, people route around it. If the google system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For google, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful google shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The google pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The google metric is route changes, eval deltas, latency movement, cost movement, and migration work. That is more useful than asking whether google "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For google, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those google numbers usually tell the truth before the happy-path dashboard does.

If the google system cannot answer those questions, I would keep the design smaller. Small google systems with honest receipts are easier to improve than large systems that only report success.
