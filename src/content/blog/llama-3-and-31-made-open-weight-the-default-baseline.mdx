Llama 3 in April 2024 was the release that made open-weight evaluation feel mandatory. Llama 3.1 in July made it harder to ignore. Meta shipped 8B and 70B models first, then added the 405B model with longer context and broader language support.

The 8B model mattered because it was practical. The 70B model mattered because it was strong enough for serious app testing. The 405B model mattered because it gave the open ecosystem a teacher model and a frontier-ish reference point that did not live entirely behind a closed API.

Compared with Llama 2, the jump felt cleaner. Better quality, better instruction behavior, better tooling support, and a bigger ecosystem response. Llama 2 had made open-weight LLMs normal. Llama 3 made them harder to exclude from a product evaluation plan.

The 3.1 release changed the baseline again. A 128K context window and a 405B model meant teams could use Llama for synthetic data generation, distillation, internal evals, and workflows where the ability to host or customize mattered more than having the single best hosted model.

I still would not call it open source without checking the license details. That distinction matters. "Open weights" gives a team control over deployment and inspection, but it is not the same as full open-source software. The practical benefit is still real. You can run it where your data lives. You can tune it. You can measure it without every call leaving your environment.

The product move I would make after Llama 3.1: keep one Llama path in the eval harness. Even if the hosted frontier model wins the hardest tasks, the open-weight model may win on cost, locality, batch processing, or governance.

The release made open models less optional. That was the real change.
