## What has to be true

For meta to be useful, the AI systems builder needs a clear model decision note. That artifact does not have to be fancy. For meta, it can be a log line, a JSON envelope, a rendered state, a small script output, or a runnable example. What matters for meta is that it explains what happened without asking the next person to reconstruct intent from vibes.

The implementation detail I care about is ownership. Who creates the artifact? Who reads it? Who is allowed to change it? In model watch, the annoying meta bugs usually live between those answers. A feature works, but the trace is missing. A model returns something valid for meta, but the caller cannot tell whether it is safe. A script exits zero for meta, but the output hides the one fact the maintainer needed.

## Implementation shape

I would make the first meta version deliberately plain. The meta system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make meta enterprise software. It makes the work inspectable.

A useful meta implementation question is where the record is created. If the meta record is created only after success, it will miss the interesting failures. I would create the meta record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The meta tradeoff is convenience versus what changed in the model versus what changed in the product surface. If the meta system is too strict, people route around it. If the meta system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For meta, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful meta shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The meta pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The meta metric is route changes, eval deltas, latency movement, cost movement, and migration work. That is more useful than asking whether meta "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For meta, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those meta numbers usually tell the truth before the happy-path dashboard does.

If the meta system cannot answer those questions, I would keep the design smaller. Small meta systems with honest receipts are easier to improve than large systems that only report success.
