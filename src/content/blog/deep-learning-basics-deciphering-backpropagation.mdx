## Implementation shape

I would make the first backpropagation version deliberately plain. The backpropagation system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make backpropagation enterprise software. It makes the work inspectable.

A useful backpropagation implementation question is where the record is created. If the backpropagation record is created only after success, it will miss the interesting failures. I would create the backpropagation record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The backpropagation tradeoff is convenience versus the exact step where a scalar, vector, matrix, or activation changes shape. If the backpropagation system is too strict, people route around it. If the backpropagation system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For backpropagation, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful backpropagation shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The backpropagation pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The backpropagation metric is whether the hand calculation, code output, and visual state agree. That is more useful than asking whether backpropagation "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For backpropagation, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those backpropagation numbers usually tell the truth before the happy-path dashboard does.

If the backpropagation system cannot answer those questions, I would keep the design smaller. Small backpropagation systems with honest receipts are easier to improve than large systems that only report success.
