Runbooks for agents get better when they include the mistakes the agent is expected to make.

The interesting part of agents is not the name. It is the system pressure around Runbooks for agents get better when they include the mistakes the agent is expected to make. Once that pressure shows up for agents, the topic stops being a feature and starts being an interface between people, models, code, and evidence.

## The unit of work

I would start by naming the unit of work for agents. In agent systems, the agents unit is usually smaller than the product surface and larger than a function call. For agents, it might be one agent step, one retrieval request, one classification decision, one model route, one sync attempt, or one evaluation case.

A useful agents unit has four properties:

- an input the system can store
- a decision the system can explain
- evidence the next component can inspect
- a boundary that says who or what is allowed to act

For agents, the boundary is file access, network access, credential use, and delegation between agents. That boundary should be visible in the agents data, not only in a policy document. If a agents request crosses the boundary, the trace should show it.

## Data flow

The simple agents flow looks like this:

~~~text
request -> context builder -> decision point -> action or answer -> verifier -> receipt
~~~

The context builder gathers the minimum useful state for agents. The agents decision point chooses a route, tool, model, policy, or user-facing behavior. The verifier checks the output against the contract. The receipt records enough evidence for review.

That receipt is the difference between a agents demo and an operable system. A demo can rely on a person watching closely. A agents system needs to explain itself after the person has moved on.

A agents request envelope might look like this:

~~~json
{
  "input": "user or system request",
  "context": ["retrieved document", "current state", "policy hint"],
  "decision": "selected route or action",
  "evidence": ["why this route was allowed"],
  "limits": { "costCents": 12, "timeoutMs": 8000 },
  "requiresApproval": false
}
~~~

The schema is deliberately plain. Fancy schemas do not save unclear agents systems. The useful agents work is deciding which fields are required, which are advisory, and which block the run when missing.

## Implementation sketch

I would implement the first agents pass as a narrow pipeline rather than a general platform. Start agents with a request envelope, make the route explicit, attach evidence, run the cheapest deterministic checks first, then escalate to semantic judgment only when the deterministic checks cannot answer the question.

~~~ts
type StepResult = {
  route: string
  output: unknown
  evidence: string[]
  deterministicChecks: string[]
  semanticReview?: string
  accepted: boolean
}
~~~

For agents, this shape matters because it keeps "the model said so" out of the acceptance path. The model can propose. The agents system still has to decide whether the proposal is usable under the current boundary.

## Failure modes

The main agents failure mode is the agent finishes a task without leaving enough evidence for a human to trust or resume it. It rarely announces itself cleanly. The model may return a good-looking answer. The UI may render. The job may exit zero. The agents failure is that the next component cannot tell whether the result should be trusted.

I would test at least five agents cases:

- missing context
- stale context
- valid output with insufficient evidence
- correct output under the wrong permission boundary
- high-confidence output that fails a deterministic check

That last agents case matters. A confident agents answer is not the same thing as a verified result. If agents touches state, policy, spend, or user trust, confidence should never be the only acceptance criterion.

## Deployment shape

I would not deploy agents as one giant agent with every tool and every permission. I would split the system by risk class. Low-risk read-only work can move quickly. State-changing work needs stronger checks. Credentialed work needs explicit scope, expiration, and audit. For agents, anything irreversible should have a human stop unless the surrounding domain has already proven a safer automatic path.

That gives the router real work to do. For agents, it should not only choose small model or big model. It should choose a risk path:

- read-only answer
- draft-only action
- reversible write
- privileged action
- escalation

Each agents path has a different budget, verifier, and approval requirement. This is the part that makes agents feel less magical and more like infrastructure. The model is still important, but the agents route determines what kind of trust the system is asking for.

## Measurement

The measurement surface for agents is approval rate, retry count, verifier disagreement, and time from tool call to accepted result. I would track agents per route or case type, not only as a global average. Global averages hide the exact class of agents problems that make these systems feel unpredictable.

A simple eval table can carry a lot of weight:

~~~text
case_id | route | expected | actual | verifier | latency_ms | cost_cents | accepted
~~~

The point is not to build an eval cathedral for agents. The point is to preserve enough agents structure that regressions can be clustered. If failures cluster around one route, fix the route. If agents failures cluster around one policy boundary, fix the boundary. If agents failures cluster around one model, stop pretending the prompt is the whole system.

## Operational concerns

The operational question for agents is how this behaves when the happy path stops being representative. What happens when context is missing? What happens when the verifier times out? What happens when the cheap agents route becomes unreliable but the expensive route would blow the budget?

For agents, I would put those answers in configuration, not in a heroic prompt. A small agents policy file that names timeouts, budgets, fallback routes, and escalation behavior will beat a clever paragraph in the system prompt every time.

## Scaling behavior

At small scale, a human can inspect weird agents cases by hand. At medium scale, the agents system needs sampling, dashboards, and escalation. At larger scale, it needs budgets and revocation. The design changes because the bottleneck changes.

Early on, the bottleneck is usually correctness. Later it becomes review capacity. After that it becomes trust: can teams change the agents system without being surprised by the downstream behavior?

For agents, I would keep the first architecture boring. Store the request envelope. Store the decision. Store the evidence. Run deterministic checks where possible. Use semantic checks where necessary. Escalate when the verifier cannot decide. That is not glamorous, but it gives the agents system a spine.

## Where I land

Agents deserves depth when it changes architecture, not when it merely sounds current. The agents implementation questions are concrete: what is the unit of work, what evidence moves with it, who can approve it, how does it fail, and what metric catches the failure before users do?

If the agents post answers those questions, it teaches something. If the agents post only says the topic matters, it is just a longer version of the headline.
