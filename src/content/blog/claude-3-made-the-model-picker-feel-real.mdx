## Implementation shape

I would make the first anthropic version deliberately plain. The anthropic system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make anthropic enterprise software. It makes the work inspectable.

A useful anthropic implementation question is where the record is created. If the anthropic record is created only after success, it will miss the interesting failures. I would create the anthropic record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## What I would measure

The anthropic metric is route changes, eval deltas, latency movement, cost movement, and migration work. That is more useful than asking whether anthropic "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For anthropic, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those anthropic numbers usually tell the truth before the happy-path dashboard does.

If the anthropic system cannot answer those questions, I would keep the design smaller. Small anthropic systems with honest receipts are easier to improve than large systems that only report success.
