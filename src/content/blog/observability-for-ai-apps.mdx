Production AI needs logs, traces, prompt records, retrieval records, and examples that can become evals.

The interesting part of observability is not the name. It is the system pressure around a team arguing about whether the model improved or merely changed. Once that pressure shows up for observability, the topic stops being a feature and starts being an interface between people, models, code, and evidence.

## The unit of work

I would start by naming the unit of work for observability. In engineering quality, the observability unit is usually smaller than the product surface and larger than a function call. For observability, it might be one agent step, one retrieval request, one classification decision, one model route, one sync attempt, or one evaluation case.

A useful observability unit has four properties:

- an input the system can store
- a decision the system can explain
- evidence the next component can inspect
- a boundary that says who or what is allowed to act

For observability, the boundary is what must be proven before the change is trusted. That boundary should be visible in the observability data, not only in a policy document. If a observability request crosses the boundary, the trace should show it.

## Data flow

The simple observability flow looks like this:

~~~text
request -> context builder -> decision point -> action or answer -> verifier -> receipt
~~~

The context builder gathers the minimum useful state for observability. The observability decision point chooses a route, tool, model, policy, or user-facing behavior. The verifier checks the output against the contract. The receipt records enough evidence for review.

That receipt is the difference between a observability demo and an operable system. A demo can rely on a person watching closely. A observability system needs to explain itself after the person has moved on.

A observability request envelope might look like this:

~~~json
{
  "input": "user or system request",
  "context": ["retrieved document", "current state", "policy hint"],
  "decision": "selected route or action",
  "evidence": ["why this route was allowed"],
  "limits": { "costCents": 12, "timeoutMs": 8000 },
  "requiresApproval": false
}
~~~

The schema is deliberately plain. Fancy schemas do not save unclear observability systems. The useful observability work is deciding which fields are required, which are advisory, and which block the run when missing.

## Implementation sketch

I would implement the first observability pass as a narrow pipeline rather than a general platform. Start observability with a request envelope, make the route explicit, attach evidence, run the cheapest deterministic checks first, then escalate to semantic judgment only when the deterministic checks cannot answer the question.

~~~ts
type StepResult = {
  route: string
  output: unknown
  evidence: string[]
  deterministicChecks: string[]
  semanticReview?: string
  accepted: boolean
}
~~~

For observability, this shape matters because it keeps "the model said so" out of the acceptance path. The model can propose. The observability system still has to decide whether the proposal is usable under the current boundary.

## Failure modes

The main observability failure mode is the system succeeds once but gives the next maintainer no way to explain it. It rarely announces itself cleanly. The model may return a good-looking answer. The UI may render. The job may exit zero. The observability failure is that the next component cannot tell whether the result should be trusted.

I would test at least five observability cases:

- missing context
- stale context
- valid output with insufficient evidence
- correct output under the wrong permission boundary
- high-confidence output that fails a deterministic check

That last observability case matters. A confident observability answer is not the same thing as a verified result. If observability touches state, policy, spend, or user trust, confidence should never be the only acceptance criterion.

## Deployment shape

I would not deploy observability as one giant agent with every tool and every permission. I would split the system by risk class. Low-risk read-only work can move quickly. State-changing work needs stronger checks. Credentialed work needs explicit scope, expiration, and audit. For observability, anything irreversible should have a human stop unless the surrounding domain has already proven a safer automatic path.

That gives the router real work to do. For observability, it should not only choose small model or big model. It should choose a risk path:

- read-only answer
- draft-only action
- reversible write
- privileged action
- escalation

Each observability path has a different budget, verifier, and approval requirement. This is the part that makes observability feel less magical and more like infrastructure. The model is still important, but the observability route determines what kind of trust the system is asking for.

## Measurement

The measurement surface for observability is reproduction time, failing example coverage, queue age, and rollback confidence. I would track observability per route or case type, not only as a global average. Global averages hide the exact class of observability problems that make these systems feel unpredictable.

A simple eval table can carry a lot of weight:

~~~text
case_id | route | expected | actual | verifier | latency_ms | cost_cents | accepted
~~~

The point is not to build an eval cathedral for observability. The point is to preserve enough observability structure that regressions can be clustered. If failures cluster around one route, fix the route. If observability failures cluster around one policy boundary, fix the boundary. If observability failures cluster around one model, stop pretending the prompt is the whole system.

## Operational concerns

The operational question for observability is how this behaves when the happy path stops being representative. What happens when context is missing? What happens when the verifier times out? What happens when the cheap observability route becomes unreliable but the expensive route would blow the budget?

For observability, I would put those answers in configuration, not in a heroic prompt. A small observability policy file that names timeouts, budgets, fallback routes, and escalation behavior will beat a clever paragraph in the system prompt every time.

## Scaling behavior

At small scale, a human can inspect weird observability cases by hand. At medium scale, the observability system needs sampling, dashboards, and escalation. At larger scale, it needs budgets and revocation. The design changes because the bottleneck changes.

Early on, the bottleneck is usually correctness. Later it becomes review capacity. After that it becomes trust: can teams change the observability system without being surprised by the downstream behavior?

For observability, I would keep the first architecture boring. Store the request envelope. Store the decision. Store the evidence. Run deterministic checks where possible. Use semantic checks where necessary. Escalate when the verifier cannot decide. That is not glamorous, but it gives the observability system a spine.

## Where I land

Observability deserves depth when it changes architecture, not when it merely sounds current. The observability implementation questions are concrete: what is the unit of work, what evidence moves with it, who can approve it, how does it fail, and what metric catches the failure before users do?

If the observability post answers those questions, it teaches something. If the observability post only says the topic matters, it is just a longer version of the headline.
