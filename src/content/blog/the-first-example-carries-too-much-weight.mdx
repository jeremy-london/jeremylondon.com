Readers build their mental model from the first runnable example, so that example has to be honest.

I would keep teaching small here. The useful part is Readers build their mental model from the first runnable example, so that example has to be honest. That is enough material for a teaching note, not a manifesto.

The teaching mistake is treating the observation as if it needs a complete theory around it. In practice, technical learning work usually gets better when the first version names the boundary and then stops. For teaching, that boundary is where explanation stops and experimentation starts.

The teaching check I would use is simple: can the next person see what changed and why it matters? If the answer is yes, the teaching post has done its job. If the answer is no, adding three more teaching paragraphs will not fix it.

The thing I would not add to teaching is a fake architecture section. A short teaching note can still be technical if it names the pressure clearly: who is acting, what state is being touched, and what evidence would make the claim falsifiable. That is enough for teaching. The rest of teaching can wait until there is a real example to inspect.

I am leaving teaching as a short note because the idea is useful at this size. A longer teaching version would need a real system diagram, a failure example, and evidence. Without those, more length would just be decoration.
