AI can generate interface pieces quickly, which makes design-system boundaries more important rather than less.

The practical version starts with AI can generate interface pieces quickly, which makes design-system boundaries more important rather than less. That gives AI ui a shape you can reason about. It also keeps the AI ui post out of slogan territory, which is where a lot of technical writing goes to die politely.

## What has to be true

For AI ui to be useful, the designer or engineer needs a clear component contract. That artifact does not have to be fancy. For AI ui, it can be a log line, a JSON envelope, a rendered state, a small script output, or a runnable example. What matters for AI ui is that it explains what happened without asking the next person to reconstruct intent from vibes.

The implementation detail I care about is ownership. Who creates the artifact? Who reads it? Who is allowed to change it? In design systems, the annoying AI ui bugs usually live between those answers. A feature works, but the trace is missing. A model returns something valid for AI ui, but the caller cannot tell whether it is safe. A script exits zero for AI ui, but the output hides the one fact the maintainer needed.

## Implementation shape

I would make the first AI ui version deliberately plain. The AI ui system should capture the request, the decision, the evidence, and the actor before it tries to be clever. That does not make AI ui enterprise software. It makes the work inspectable.

A useful AI ui implementation question is where the record is created. If the AI ui record is created only after success, it will miss the interesting failures. I would create the AI ui record before the risky step, update it as the work moves, and leave enough state behind that a failed run can be replayed without asking someone to remember what was on their screen.

## Tradeoffs

The AI ui tradeoff is convenience versus which decisions belong in tokens, components, product copy, or one-off layout. If the AI ui system is too strict, people route around it. If the AI ui system is too loose, it becomes impossible to tell whether a successful run was actually acceptable. For AI ui, I would bias toward a small explicit contract first, then widen it only when the examples prove the narrower version is too painful.

That contract can be boring. A useful AI ui shape is:

~~~ts
type WorkRecord = {
  input: string
  decision: string
  evidence: string[]
  actor: string
  accepted: boolean
}
~~~

The exact fields change by system. The AI ui pattern matters more than the names: capture the input, the decision, the evidence, the actor, and whether the result was accepted.

## What I would measure

The AI ui metric is token reuse, unsupported variants, visual regression count, and handoff ambiguity. That is more useful than asking whether AI ui "worked." Worked for whom? At what cost? Under which boundary? With what recovery path?

For AI ui, I would also watch the boring negative space: how many runs have missing evidence, how many require manual reconstruction, and how often the system reports success while the human rejects the result. Those AI ui numbers usually tell the truth before the happy-path dashboard does.

If the AI ui system cannot answer those questions, I would keep the design smaller. Small AI ui systems with honest receipts are easier to improve than large systems that only report success.
