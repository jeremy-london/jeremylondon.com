Offline capability is not a storage trick. It is a promise about what the user can trust when the network disappears.

I would keep offline first small here. The useful part is software that keeps working when the network stops helping. That is enough material for a offline first note, not a manifesto.

The offline first mistake is treating the observation as if it needs a complete theory around it. In practice, local-first software work usually gets better when the first version names the boundary and then stops. For offline first, that boundary is what must be true before the server is reachable.

The offline first check I would use is simple: can the next person see what changed and why it matters? If the answer is yes, the offline first post has done its job. If the answer is no, adding three more offline first paragraphs will not fix it.

The thing I would not add to offline first is a fake architecture section. A short offline first note can still be technical if it names the pressure clearly: who is acting, what state is being touched, and what evidence would make the claim falsifiable. That is enough for offline first. The rest of offline first can wait until there is a real example to inspect.

I am leaving offline first as a short note because the idea is useful at this size. A longer offline first version would need a real system diagram, a failure example, and evidence. Without those, more length would just be decoration.
