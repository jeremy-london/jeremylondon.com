Offline capability is not a storage trick. It is a promise about what the user can trust when the network disappears. The useful version is practical: software that keeps working when the network stops helping.

## What Changes

For offline first, I would start from the constraint, not the announcement. The constraint is software that keeps working when the network stops helping, and the surrounding question is how offline behavior, sync, local state, and private work changes in practice.

## Where It Fails

The thing I would watch for with offline first is quieter than a crash: the app works online but loses the user once the network or device state changes. That is the kind of break that makes a offline first post feel padded even when the topic is legitimate.

## The Check

My bar for offline first is the rule for what happens before the server is reachable. If the offline first post cannot show that, it should stay short and honest.

That is where I would leave it. Keep the offline first claim narrow, make the tradeoff visible, and let the example do the work.
