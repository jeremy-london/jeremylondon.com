A personal site needs occasional pruning because old work keeps speaking after the context disappears.

I would keep portfolio small here. The useful part is a personal site needs occasional pruning because old work keeps speaking after the context disappears. That is enough material for a portfolio note, not a manifesto.

The portfolio mistake is treating the observation as if it needs a complete theory around it. In practice, personal systems work usually gets better when the first version names the boundary and then stops. For portfolio, that boundary is how much friction belongs at capture time versus review time.

The portfolio check I would use is simple: can the next person see what changed and why it matters? If the answer is yes, the portfolio post has done its job. If the answer is no, adding three more portfolio paragraphs will not fix it.

The thing I would not add to portfolio is a fake architecture section. A short portfolio note can still be technical if it names the pressure clearly: who is acting, what state is being touched, and what evidence would make the claim falsifiable. That is enough for portfolio. The rest of portfolio can wait until there is a real example to inspect.

I am leaving portfolio as a short note because the idea is useful at this size. A longer portfolio version would need a real system diagram, a failure example, and evidence. Without those, more length would just be decoration.
